# 🏘️ Mezha

> Ефективний веб-застосунок на **FastAPI** для автоматизованого аудиту земельних та майнових реєстрів територіальних громад. Система виявляє невідповідності (тіньові активи) та проводить їх через структурований рольовий процес перевірки.

**Життєвий цикл аномалії:** `АДМІНІСТРАТОР ➔ ВОЛОНТЕР ➔ ІНСПЕКТОР`

---

## 🎯 Головні можливості системи

* **Розумний імпорт:** Завантаження реєстрів землі та нерухомості (CSV/XLSX).
* **Глибокий аудит:** Пошук розбіжностей за допомогою Fuzzy Matching та збагачення даних через AI.
* **Класифікація:** Генерація аномалій із маркуванням `RED` (критичні) або `GREEN` (потенційні інвест-активи).
* **Прозорий Workflow:** Покрокова валідація знахідок на місцях.
* **Аудит-контроль:** Незмінний журнал подій (`audit_logs`) для відстеження всіх дій користувачів.

---

## 👥 Ролі та права доступу

Система має чітке розмежування доступу:

* **ADMIN:** Завантажує реєстри, запускає аудит, керує статусом `PENDING_ADMIN`, має доступ до повної статистики та журналу логів.
* **VOLUNTEER:** Бачить пул доступних задач (`RED` аномалії), бере їх у роботу, виїжджає на місце та завантажує фото-звіт із коментарем.
* **INSPECTOR:** Опрацьовує задачі у статусі `PENDING_INSPECTOR`, фінально підтверджуючи або відхиляючи звіт волонтера.

## 🚦 Статуси аномалій (Workflow)

Повний шлях аномалії від виявлення до закриття:

1. **`PENDING_ADMIN`** — Створена системою, очікує первинного рішення Адміністратора.
2. **`NEW`** — Підтверджена Адміністратором, відправлена у пул Волонтерів.
3. **`IN_WORK`** — Взята в роботу конкретним Волонтером.
4. **`PENDING_INSPECTOR`** — Волонтер надіслав звіт (фото+коментар), очікує перевірки.
5. **`RESOLVED`** — Інспектор підтвердив кейс (успішне завершення).
6. **`DISMISSED`** — Відхилена Інспектором.

## 🚀 Швидкий старт (Docker)

Це рекомендований спосіб розгортання, який автоматично піднімає БД, застосовує міграції та створює демо-користувачів.

**Крок 1.** Створіть файл конфігурації з прикладу:
```bash
cp .env.example .env
```

**Крок 2.** Запустіть контейнери:
```bash
docker compose up --build
```

**Крок 3.** Відкрийте застосунок у браузері:
* **UI:** `http://localhost:8000/`
* **Swagger API:** `http://localhost:8000/docs`

---

## 💻 Локальний запуск (Без Docker)

**Крок 1.** Підготуйте PostgreSQL та створіть БД (напр., `assetvision`).

**Крок 2.** Налаштуйте `.env` файл (вкажіть локальний `DATABASE_URL`).

**Крок 3.** Встановіть залежності у віртуальне середовище:
```bash
python -m venv .venv
source .venv/bin/activate
pip install -r requirements.txt
```

**Крок 4.** Виконайте міграції та заповніть БД початковими даними:
```bash
alembic upgrade head
python scripts/seed.py
```

**Крок 5.** Запустіть сервер:
```bash
uvicorn app.main:app --reload
```

## ⚙️ Конфігурація (.env)

Обов'язкові змінні для роботи системи:
* `DATABASE_URL` — рядок підключення до PostgreSQL.
* `SECRET_KEY` та `REFRESH_SECRET_KEY` — ключі шифрування JWT.

Додаткові важливі параметри:
* `OPENAI_API_KEY` — ключ для роботи AI-модуля збагачення даних.
* `MEDIA_ROOT` — директорія для збереження фото-звітів волонтерів.

## 📡 Документація API

Усі запити до API вимагають авторизації (окрім публічних роутів). Підтримується Bearer токен та `access_token` у cookie (для HTMX).

### 🖥️ Інтерфейс (UI)
| Метод | Шлях | Доступ | Опис |
| --- | --- | --- | --- |
| `GET` | `/` | Публічний | Головна сторінка |
| `GET` | `/login` | Публічний | Сторінка авторизації |
| `GET` | `/admin` | `ADMIN` | Дашборд адміністратора |
| `GET` | `/volunteer` | `VOLUNTEER` | Кабінет волонтера |
| `GET` | `/inspector` | `INSPECTOR` | Кабінет інспектора |

### 🔑 Авторизація (`/api/auth`)
| Метод | Шлях | Опис |
| --- | --- | --- |
| `POST` | `/api/auth/login` | Повертає токени, ставить cookie, робить `HX-Redirect` на профіль |
| `POST` | `/api/auth/refresh` | Оновлює пару токенів за допомогою `refresh_token` |
| `POST` | `/api/auth/logout` | Видаляє cookie, редірект на головну сторінку |

### 🧠 Аудит та Дані (`/data` - тільки `ADMIN`)
| Метод | Шлях | Опис |
| --- | --- | --- |
| `POST` | `/data/upload-registers` | Завантаження файлів `land_file` та `real_estate_file` |
| `POST` | `/data/run-audit` | Запуск алгоритму співставлення та генерація аномалій |
| `GET` | `/api/audit-logs` | Журнал дій по системі |

### 🔍 Аномалії (`/api/anomalies`)
| Метод | Шлях | Доступ | Опис |
| --- | --- | --- | --- |
| `GET` | `/api/anomalies/stats` | `ADMIN` | Статистика по статусах аномалій |
| `GET` | `/api/anomalies/pending-admin` | `ADMIN` | Черга на перевірку адміністратором |
| `POST` | `/{id}/admin-decision` | `ADMIN` | Підтвердження або відхилення аномалії |
| `GET` | `/api/anomalies/pool` | `VOLUNTEER` | Запит пулу доступних задач |
| `POST` | `/{id}/take` | `VOLUNTEER` | Переведення задачі у статус `IN_WORK` |
| `POST` | `/{id}/volunteer-report` | `VOLUNTEER` | Відправка фото та коментаря з місця |
| `GET` | `/pending-validation` | `INSPECTOR` | Черга звітів від волонтерів на перевірку |
| `POST` | `/{id}/report` | `INSPECTOR` | Фінальне рішення по звіту волонтера |
